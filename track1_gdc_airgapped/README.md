# GDC Air-Gapped Inference Gateway (Track 1)

This directory contains the foundational, production-ready implementation for **Track 1 of the Google Sovereign AI Portfolio: GDC Air-Gapped Inference**.

The architecture establishes a zero-trust model boundary inside a Google Distributed Cloud (GDC) hardware rack, preventing parameters and query responses from escaping local physical nodes by enforcing local authentication and routing external inference traffic exclusively through secure overlay network overlays.

## LiteLLM Proxy & Network Architecture

To keep the GDC enclave air-gapped while enabling distributed inference, all outgoing model request execution is routed exclusively over a **Tailscale overlay network**.

### 1. Configuration Parameters
The gateway uses the following environment variables to authenticate and route requests to the Proxmox environment:
* `LITELLM_PROXY_URL`: Target endpoint for the Proxmox-hosted LiteLLM router (Default: `http://100.116.70.21:4000/v1/chat/completions`).
* `LITELLM_API_KEY`: Authentication credential for the sovereign proxy (Default: `sk-sovereign-gateway-2026`).

### 2. Tailscale Subnet Routing
* The GDC rack blocks all public internet destinations (`0.0.0.0/0`) at the firewall layer.
* Traffic to the Carrier-Grade NAT (CGNAT) Tailscale overlay range (`100.64.0.0/10`) on TCP port `4000` is explicitly whitelisted.
* This architecture ensures all model interactions are encrypted in transit and confined strictly to the approved Tailscale mesh, precluding direct public internet access.

## Deployment & Setup Instructions

### 1. Build and Run the Enclave Container
In the secure host environment, build the gateway image locally and start the container with the environment configurations:
```bash
# Build the sovereign docker image
docker build -t gdc-gemma3-inference:latest .

# Run the container directing to the Tailscale LiteLLM proxy
docker run -d -p 8000:8000 \
  --name sovereign-gateway \
  -e LITELLM_PROXY_URL="http://100.116.70.21:4000/v1/chat/completions" \
  -e LITELLM_API_KEY="sk-sovereign-gateway-2026" \
  gdc-gemma3-inference:latest
```

### 2. Attestation & Execution Flow
Every inbound request requires two security validation headers:
* `X-User-Role`: Set to a permitted role (e.g., `Admin` or `DataAnalyst`).
* `X-Attestation-Token`: The signed OIDC token generated by the enclave's local hardware attestation agent.
